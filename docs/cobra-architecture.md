# Cobra Architecture

## Overview

rift uses [Cobra](https://github.com/spf13/cobra) as its CLI framework. Cobra models a CLI as a tree of commands — one root with any number of subcommands, each independently defining its own arguments, flags, and behaviour.

## How the tree is wired

```
main.go
  └── cmd.Execute()          ← single entry point, always called on root

rootCmd                      ← rift [nickname] / rift --memorize <nickname>
  └── awakenCmd              ← rift awaken
```

`main.go` never imports individual subcommand files. It only calls `cmd.Execute()`. Subcommands register themselves onto `rootCmd` via their own `init()` function, which Go runs automatically at package load time.

Adding a new subcommand means creating a new file in `cmd/`, defining the command, and calling `rootCmd.AddCommand(...)` in `init()`. Nothing else needs to change.

## Root command

`rootCmd` handles two things depending on which flag/arg is present:

| Input | Behaviour |
|---|---|
| `rift <nickname>` | look up path, print `cd <path>` to stdout for shell eval |
| `rift --memorize <nickname>` | save current directory under the nickname |
| `rift` (no args, no flag) | print help |

## File structure

Each file in `cmd/` owns one command — its definition, its flags, and its `init()` registration.

```
cmd/
  root.go      — rootCmd definition, Execute(), and root-level flags
  setup.go     — setupCmd + its own flags
  remove.go    — removeCmd + its own flags
  list.go      — listCmd + its own flags
```

Go allows multiple `init()` functions across files in the same package, so each file self-registers without coordinating with others. `main.go` stays a one-liner forever.

## Defining a subcommand

```go
// cmd/remove.go
var removeCmd = &cobra.Command{
    Use:   "remove <nickname>",
    Short: "Remove a saved nickname",
    Args:  cobra.ExactArgs(1),
    RunE: func(cmd *cobra.Command, args []string) error {
        // args[0] is the nickname
        return nil
    },
}

func init() {
    rootCmd.AddCommand(removeCmd)
}
```

## Argument constraints

Cobra provides built-in validators for `Args`:

| Validator | Meaning |
|---|---|
| `cobra.ExactArgs(n)` | exactly n positional args |
| `cobra.MinimumNArgs(n)` | at least n args |
| `cobra.MaximumNArgs(n)` | at most n args |
| `cobra.RangeArgs(min, max)` | between min and max args |
| `cobra.ArbitraryArgs` | any number, including zero |

## Flags

### Local vs persistent

```go
// Local — only available on this specific command
myCmd.Flags().BoolP("force", "f", false, "force removal")

// Persistent — available on this command and all subcommands beneath it
rootCmd.PersistentFlags().BoolP("verbose", "v", false, "verbose output")
```

Flags can appear anywhere in the command — `rift merge a b --no-ff c` works fine. Cobra separates flags from positional args regardless of order.

### Each command owns its own flags

Flags are defined and registered inside the same file as the command that uses them. There is no centralised flags file.

```go
// cmd/remove.go
package cmd

var forceRemove bool

var removeCmd = &cobra.Command{
    Use:  "remove <nickname>",
    Args: cobra.ExactArgs(1),
    RunE: func(cmd *cobra.Command, args []string) error {
        // forceRemove is already populated by cobra via BoolVarP binding
        // use forceRemove and args[0]
        return nil
    },
}

func init() {
    removeCmd.Flags().BoolVarP(&forceRemove, "force", "f", false, "skip confirmation")
    rootCmd.AddCommand(removeCmd)
}
```

Root-level flags (those on `rootCmd`) live in `root.go` alongside the command they belong to.

### Flag relationships

**`MarkFlagsRequiredTogether`** enforces that if any one of the listed flags is provided, all must be provided. This is symmetric — it cuts both ways:

```go
rootCmd.MarkFlagsRequiredTogether("foo", "bar")
// --foo alone  → error
// --bar alone  → error
// --foo --bar  → ok
```

This is only appropriate when all paired flags are equally mandatory together.

**For optional extensions** — where one flag is valid alone but another only makes sense alongside it — use manual validation in `RunE` instead:

```go
// --memorize works alone, --command is only valid with --memorize
if cmd.Flags().Changed("command") && !cmd.Flags().Changed("memorize") {
    return fmt.Errorf("--command requires --memorize")
}
```

`cmd.Flags().Changed(name)` returns true only when the user explicitly passed the flag, regardless of its value — checking the string value directly would incorrectly pass for `--command ""`.

This gives you full control over the relationship without the symmetric constraint that `MarkFlagsRequiredTogether` imposes.

## Stdout discipline

Because `rift <nickname>` emits a raw `cd <path>` to stdout for the shell wrapper to `eval`, all other output — help text, error messages, subcommand output — must go to stderr.

This is enforced at the root level:

```go
rootCmd.SetOut(os.Stderr)
rootCmd.SetErr(os.Stderr)
```

Subcommands inherit this setting. The rule: **never write to stdout unless you are intentionally emitting a shell command to eval**.
